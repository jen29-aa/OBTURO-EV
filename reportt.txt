ABSTRACT
The Obturo-EV Intelligent Electric Vehicle (EV) Charger Booking and Management System is an
innovative platform delivered through a Flutter-based Android mobile application and a Django￾served web application, both powered by a single shared backend. The system addresses major
challenges faced by EV users such as limited charger accessibility, booking conflicts, and unre￾liable status updates. Obturo-EV enables users to search, compare, and reserve charging stations
based on personalized preferences including price, distance, waiting time, and charging speed.
By leveraging Multi-Attribute Decision-Making (MADM) methods and the TOPSIS algorithm,
the system provides ranked recommendations that dynamically adapt to user priorities. It fur￾ther incorporates route-based discovery, conflict-free booking, and smart waitlist management to
ensure guaranteed charger availability. The Android application delivers real-time WebSocket
slot updates, Firebase Cloud Messaging push notifications, and an interactive map-based station
discovery interface, while the web application additionally provides an administrative analytics
dashboard for platform monitoring. The inclusion of Peer-to-Peer (P2P) charger listings enhances
network scalability and optimizes infrastructure utilization. Built with robust JWT authentication,
penalty-based policy enforcement, and real-time data handling via Django Channels, Obturo-EV
ensures fairness, reliability, and data integrity, ultimately promoting sustainable transportation
through a unified and user-centric EV charging experience across both mobile and web.

Chapter 1
INTRODUCTION
In today’s rapidly advancing world, the rise of electric vehicles (EVs) represents a major step
toward sustainable and eco-friendly transportation . However, despite the growing adoption of
EVs, one of the major challenges faced by users is the limited availability and accessibility of
charging infrastructure . Traditional EV charging systems rely mainly on static filtering methods,
such as proximity or price, which fail to address dynamic user needs like waiting time, budget, or
charging speed
To overcome these challenges, the Obturo-EV Intelligent Charger Booking and Management Sys￾tem has been proposed. This web-based application provides EV users with an intelligent plat￾form to discover, compare, and reserve charging stations efficiently. It integrates Multi-Attribute
Decision-Making (MADM) techniques and the TOPSIS (Technique for Order of Preference by
Similarity to Ideal Solution) algorithm to deliver personalized and optimized recommendations
based on user preferences such as waiting time, price, distance, and charging duration.
By combining real-time data processing, smart booking, and conflict handling, Obturo-EV ensures
a seamless charging experience that minimizes waiting time, prevents double booking, and pro￾motes fairness through queue management and penalty enforcement. Ultimately, the system aims
to make EV charging more reliable, convenient, and user-centric, encouraging wider adoption of
electric mobility .
1.1 Problem Statement
The rapid growth of electric vehicles has not been matched by an equally efficient and intelligent
charging network. Existing EV charging platforms are limited to basic filters such as nearest loca￾tion or static cost, without considering dynamic user requirements or real-time station availability.
This often leads to long waiting times, inefficient resource utilization, and double booking
conflicts. Moreover, the absence of unified data from multiple charging operators results in poor
reliability and lack of transparency in slot availability.

1.2 Objective
1. Personalized Station Recommendation: Implement the TOPSIS (Technique for Order of
Preference by Similarity to Ideal Solution) multi-criteria decision-making algorithm to rank nearby
charging stations according to user-defined preference weights for six criteria — distance from
current location, price per kWh, waiting time, charging speed (kW), available slots, and estimated
charging duration. The ranking adapts dynamically to each user’s priorities, ensuring the most
suitable station is always surfaced first.
2. Guaranteed Conflict-Free Booking: Prevent overbooking through server-side time-window
overlap detection using Django ORM range filters. Every booking request is validated against all
existing active bookings for the selected station and time window before confirmation, ensuring
no two users are ever assigned the same slot simultaneously.
3. Real-Time Slot Availability: Provide live slot count updates to all connected users through
Django Channels WebSocket broadcasting. When any booking is created or cancelled, the updated
slot count is pushed instantly to every Flutter and browser client viewing that station, eliminating
the need for manual refresh or periodic polling.
4. Automated Waitlist Management: Implement a position-based waitlist queue that automati￾cally promotes the next waiting user when a slot becomes available — either through a cancellation
or a completed booking detected by the background scheduler. Users are notified of their queue
position, estimated wait time, and slot promotion via push notifications.
5. Route-Based Charging Station Discovery: Integrate mapping APIs to identify and display
relevant charging stations along the user’s planned travel route. Stations are filtered by perpen￾dicular distance to the route polyline and ranked using the TOPSIS algorithm to surface the best
charging stops for the journey.
6. Peer-to-Peer Charger Sharing Marketplace: Enable private charger owners to list their per￾sonal charging infrastructure on the platform and accept booking requests from other EV drivers.
The P2P module implements a complete two-sided marketplace with owner approval or rejection,
overlap conflict detection, and real-time push notifications to both parties at every step.
7. Smart EV-Aware Filtering: Maintain an EV car catalogue with battery capacity, maximum
DC and AC charging power, connector type, and WLTP range. Use the authenticated user’s reg￾istered vehicle profile to filter stations by compatible connector type and recommend DC fast
charging or AC slow charging based on the car’s specifications.

8. Automated Booking Lifecycle Management: Deploy a background scheduler (APScheduler)
that runs every minute to automatically mark expired bookings as completed, free the associ￾ated slot, and trigger waitlist promotion. The scheduler also dispatches 10-minute pre-charging
reminder notifications, reducing no-shows and improving station utilization.
9. Admin Analytics and Station Management: Provide a staff-only analytics dashboard expos￾ing platform-wide KPIs including total users, total bookings, estimated revenue, system utilization
rate, top stations by booking volume, and daily and monthly revenue breakdowns, enabling data￾driven operational decisions.
1.3 Scope
1. The system provides secure user registration with field-level validation — enforcing unique
usernames, unique email addresses, minimum password strength, and email format checks — fol￾lowed by JWT-based authentication using short-lived access tokens and long-lived refresh tokens,
ensuring all API endpoints are protected against unauthorized access.
2. The platform maintains a comprehensive EV car catalogue and allows users to register their
specific vehicle model. Battery capacity, maximum DC and AC charging power, connector type,
and WLTP range are stored per vehicle, enabling the system to recommend the most suitable
charger type and filter incompatible stations from discovery results.
3. The booking module guarantees conflict-free reservations through automated server-side over￾lap detection using time-window range queries against all active bookings. If a requested time
slot is unavailable, the user is automatically placed on a position-based waitlist with a computed
estimated wait time derived from recent booking durations at that station.
4. The system provides real-time slot availability updates to all connected clients through Django
Channels WebSocket broadcasting backed by a Redis channel layer. Every booking or cancella￾tion event triggers an instant slot count update pushed simultaneously to all Flutter app and web
browser clients viewing the affected station, without any client-side polling.
5. The intelligent station discovery module uses the TOPSIS multi-criteria decision algorithm to
rank nearby stations based on six weighted criteria: available slots, charging power, waiting time,
estimated charging duration, price per kWh, and distance. User-defined preference weights are
normalized and applied through a NumPy-vectorised matrix computation on the backend, with a
local fallback scoring implementation in the Flutter app for offline resilience.

The Authentication module spans both the Flutter app and the backend. On the Flutter side, the
login and registration screens call ApiService with user credentials. The backend SignupView
enforces field validation and password hashing; the LoginView returns a JWT access and refresh
token pair via SimpleJWT. The Flutter ApiService stores tokens through StorageService and
injects the bearer token header into all subsequent requests automatically. The module also handles
EV car profile selection, smart DC vs AC recommendation based on the user’s car specifications,
and FCM device token registration on login.
Module 2 — Station Discovery Module
The Station Discovery module in the Flutter app uses StationService to fetch all stations, nearby
stations filtered by the Haversine formula, text search results, station details, smart car-filtered lists,
and user favourites. The ChargingStation Dart model holds all station fields including optional
runtime values such as distanceKm computed on the client. The backend ChargingStation
Django model stores 25 fields covering name, coordinates, charger type, connector type, power
rating, slot counts, pricing, facilities, amenity flags, and operating hours. Proxy endpoints for
Nominatim geocoding and OSRM turn-by-turn routing are also exposed for the Flutter map view.
Module 3 — Smart TOPSIS Ranking Module
The Smart Ranking module is the distinguishing feature of Obturo. The Flutter SmartRankingScreen
exposes six weight sliders (available slots, power, waiting time, charging time, price per kWh,
distance) that store values on a 1–10 scale in RankingProvider. When the user triggers rank￾ing, RankingProvider.fetchRanking() calls RankingService.getSmartRanking() with the
user’s GPS coordinates and normalized weights. The backend TOPSIS endpoint builds a decision
matrix, runs the TOPSIS algorithm using NumPy, and returns stations sorted by descending score.
If the call fails, the provider executes a local fallback using station data from StationService,

3.3.2 Database Design
The Obturo database schema consists of eleven tables managed through Django’s ORM. The
auth user table (Django built-in) stores core identity fields. The accounts car table holds
the EV catalogue with battery capacity, max DC/AC power, connector type, and WLTP range.
The accounts usercar table is a one-to-one extension linking a user to their chosen car. The
accounts devicetoken table stores FCM tokens registered by the Flutter app on login, with
a unique constraint on the token string. The stations chargingstation table is the cen￾tral entity with 25 columns. The stations booking table has foreign keys to auth user and
stations chargingstation with start time, end time, and status. The stations waitlist table
adds a position integer and expires at timestamp with a unique together constraint on (user, sta￾tion). The stations peercharger and stations peerbooking tables implement the P2P mar￾ketplace. The stations stationrating table enforces one review per user per station. The
stations stationreport table includes an expires at field used to auto-purge crowd reports
older than 2 hours.
3.3.3 Interface Design
The Flutter application follows a layered interface design. The Navigation layer in lib/main.dart
registers all named routes — login, registration, home, station list, station detail, smart ranking,
route planner, booking, and P2P screens. The State Binding layer uses MultiProvider at the app
root to initialize all providers once; individual screens access providers via Provider.of<T>(context)
or Consumer<T> widgets that rebuild only the affected subtree when state changes.
All backend API interfaces follow RESTful conventions with JSON bodies. The Flutter ApiService
singleton applies the Authorization: Bearer <token> header to all authenticated requests.
Error responses use a consistent JSON structure with an error key; the ApiService interprets
status codes 401, 403, 404, and timeout exceptions and rethrows typed errors that provider classes
catch and expose as user-friendly error messages in the UI. All datetime values are converted to
the local timezone (Asia/Kolkata) by the backend before being returned, so the Flutter app always
displays correctly formatted local times without additional client-side conversion. The WebSocket
interface sends JSON messages with initial data on connect and slots update on every slot
change broadcast.

3.4 Implementation Details
Implementing OBTURO-EV involves careful coordination between the Flutter frontend and the
Django backend across the booking, ranking, routing, notification, and real-time layers.
The Flutter application follows a strict unidirectional data flow. The user interacts with a UI
screen, which calls a provider method. The provider calls a service. The service uses ApiService
to perform network I/O. The backend returns data. JSON is converted into Dart model objects.
The provider updates its state and calls notifyListeners(). Consumer widgets rebuild auto￾matically. This flow applies consistently to station discovery, TOPSIS ranking, booking, route
planning, and P2P operations.
The ApiService singleton is initialized once and reused across all services. It adds the Content-Type:
application/json header to every request and retrieves the stored JWT token from StorageService
to inject as a bearer token when required. It implements a centralized timeout and catches 401, 403,
and 404 HTTP errors, allowing service classes to simply rethrow exceptions and letting provider
classes interpret them as user-facing messages.
The smart ranking implementation in RankingProvider is particularly robust. It normalizes user
weights — stored on a 1–10 integer scale from the SmartRankingScreen sliders — to decimal
percentages before sending them to the backend. If the remote TOPSIS call returns a 404 or
network error, the provider immediately executes a local fallback: it calls StationService to
fetch all stations, computes Haversine distances, filters to a local radius, normalizes benefit criteria
(power, available slots — higher is better) and cost criteria (price, distance — lower is better)
independently, computes a weighted composite score, sorts descending, and stores the top results.
A boolean flag isFallback is set to true so the UI can display an appropriate offline banner to the
user.
On the backend, the booking engine in create booking follows a three-gate pattern. The first
gate queries UserPenalty and rejects blocked users with HTTP 403, capping blocks at 5 min￾utes. The second gate uses Django ORM filters start time lt=end and end time gt=start
to count overlapping active bookings; if this equals total slots, the user is placed on the waitlist.
The third gate creates the booking, decrements available slots, calls broadcast slot update
(which uses async to sync to push a WebSocket group message to all connected Flutter clients
viewing that station), and dispatches confirmation email and FCM push in daemon threads.
The RoutePlannerProvider sends the full list of GPS coordinate points for a planned route to
RoutePlannerService, which submits them to two backend endpoints simultaneously — sta￾tions along the route and best charging stops. The backend computes perpendicular distances

from each station to each polyline segment and filters within the specified radius. The provider

Finally, email notifications are sent using Django’s send mail with html message to deliver

# Utility: Haversine Distance
# -------------------------
def calculate_distance(lat1, lon1, lat2, lon2):
R = 6371 # km
dlat = math.radians(lat2 - lat1)
dlon = math.radians(lon2 - lon1)
a = (
math.sin(dlat / 2) ** 2
+ math.cos(math.radians(lat1))
* math.cos(math.radians(lat2))
* math.sin(dlon / 2) ** 2
)
return R * 2 * math.atan2(math.sqrt(a), math.sqrt(1 - a))
